# 사업계획서 에이전트 운영 규칙 (Rules)

> 이 파일의 규칙은 모든 스킬·워크플로보다 **우선 적용**됩니다.

---

## 🔴 절대 규칙 (절대 어기면 안 됨)

### R-01: 프로젝트 우선 확인
- 사업계획서 관련 모든 작업 시작 전, `projects/` 폴더를 먼저 확인한다.
- 신규/기존 프로젝트 여부를 사용자에게 명확히 안내하고 선택을 받는다.
- **절대로 프로젝트 폴더 없이 바로 섹션 작성을 시작하지 않는다.**

### R-02: 출처 없는 수치 금지
- 모든 시장 규모, CAGR, 통계 수치는 반드시 출처 태그 포함.
  - 형식: `* 출처: 기관명(연도)`
- 출처를 찾을 수 없는 수치는 작성하지 않고 사용자에게 확인 요청.

### R-03: 목차 정합성 의무
- UI 컴포넌트 생성 전 `references/toc.md` (프로젝트 내) 또는 루트 `목차.md`를 반드시 확인.
- 공식 목차와 1:1 매핑이 되지 않는 섹션은 생성 금지 → 사용자에게 먼저 보고.

### R-04: 사업비 집행 규정 준수
- 사업비 관련 내용 작성 시 `fund-execution-guidelines` 스킬 기준 반드시 적용.
- 인건비, 개인 용도 물품 항목 절대 포함 불가.

### R-05: 파일 저장 위치 규정
| 파일 유형 | 저장 위치 |
|---------|---------|
| 섹션 텍스트 초안 | `projects/{프로젝트}/sections/*.md` |
| React UI 컴포넌트 | `projects/{프로젝트}/ui/*.jsx` |
| 시장 조사 데이터 | `projects/{프로젝트}/context/market-data.md` |
| 참조 문서 (HWP 추출 등) | `projects/{프로젝트}/references/` |
| 프로젝트 메타데이터 | `projects/{프로젝트}/project.json` |

## 🟡 작동 및 작성 품질 규칙

### R-06: 개조식 문체 원칙
- 사업계획서 본문은 개조식(bullet point 중심) 작성.
- 불필요한 영어 비즈니스 용어 최소화:
  - BM → 수익 모델 | Pain Point → 핵심 문제 | GTM → 시장 진입 전략
  - Scale-up → 성장 단계 | Target → 핵심 고객 | KPI는 그대로 사용 가능

### R-07: 섹션 순차 작성 원칙
- 한 번에 전체 섹션을 작성하지 않는다.
- 1개 섹션 완성 → 사용자 검토 → 다음 섹션 순서로 진행.
- 단, 사용자가 명시적으로 "전체 작성"을 요청한 경우 예외.

### R-08: 스킬 자동 트리거 규칙
| 사용자 발화 패턴 | 자동 활성 스킬 |
|--------------|-------------|
| "사업계획서 [작성/써줘/시작]" | `business-plan-writer` + `conversation-init` 워크플로 |
| "[섹션명] 섹션 작성" | `business-plan-writer` (해당 섹션 가이드 로드) |
| "[섹션명] UI 만들어줘" | `hwp-styled-ui` + `official-toc-mapper` |
| "평가해줘 / 점수 줘" | `business-plan-evaluator` |
| "HWP 읽어줘 / 분석해줘" | `hwp-extractor` |
| "사업비 집행 / 예산 계획" | `fund-execution-guidelines` |
| "/business-plan-agent" | 전체 `business-plan-agent` 워크플로 실행 |
| "/conversation-init" | 프로젝트 초기화 및 컨텍스트 복원 파이프라인 |

## 🎨 UI 생성 규칙 (HWP 스타일)

### R-09: HWP 스타일 기본 원칙
- 모노크롬(흑백) 원칙: 컬러는 완전 금지 (Recharts 포함)
- `border-radius` 0 (둥근 모서리 금지)
- `lucide-react` 아이콘 등 외부 아이콘 라이브러리 사용 금지 (단순 텍스트/기호 사용)
- 모든 수치에 출처 표기 (`* 출처:`)
- 컴포넌트명 규칙: `{섹션명식별자}V{버전번호}.jsx` (예: `ProblemRecognitionV1.jsx`)

### R-10: 레이아웃 패턴 규칙
- **표(Table) 디자인**: 상단/하단은 2px 굵은 테두리, 내부 셀은 1px 얇은 테두리 적용.
- **데이터 컴포넌트**: 가급적 한글의 '글상자(Text Box)' 형태로 레이아웃(Box 기반 구조) 구성.
- **프로세스 흐름**: 가로 정렬된 상자 간 단순 텍스트 화살표(`→`)를 사용하여 연결.

### R-11: 빈 문서(Placeholder) 처리 규칙
- 원본 텍스트에 내용이 없거나 "작성 예정"으로 명시된 부분에서 절대 내용을 지어내지(hallucinate) 않음.
- 파선(dashed) 테두리 상자에 "작성 예정" 또는 해당 섹션명을 명시한 빈 Placeholder UI만을 생성.

## 🟢 컨텍스트 관리 규칙

### R-12: 최소 로드 원칙
- 현재 작업 섹션 관련 파일만 로드 (다른 섹션 컴포넌트 동시 로드 금지)
- references 파일은 한 번에 1개만 로드
- 구버전 컴포넌트 (V1~V7) 열람은 명시적 비교 요청 시에만 허용

### R-13: 프로젝트 상태 즉시 갱신
- 섹션 초안 완성 시 → `sections/{섹션}.md` 파일에 즉시 저장
- UI 컴포넌트 생성 시 → `project.json`의 `ui_components` 배열 업데이트
- 섹션 완료 시 → `project.json`의 `completed_sections` 배열 업데이트

## 📋 프로젝트 상태 머신

```text
[미시작] ──→ [진행중] ──→ [검토중] ──→ [완료]
                ↑                          |
                └──────── [수정중] ←───────┘
```

| 상태 | 정의 | 다음 행동 |
|------|------|---------|
| `미시작` | 섹션 초안 없음 | 작성 시작 |
| `진행중` | 초안 작성 중 | 계속 작성 |
| `검토중` | Critic 평가 완료, 수정 대기 | 사용자 피드백 반영 |
| `완료` | UI 포함 최종 완성 | 다음 섹션으로 이동 |
| `수정중` | 완료 후 재수정 요청 | 해당 파일만 수정 |

## 🔧 스크립트 사용 규칙

### 프로젝트 초기화 스크립트
```powershell
# 새 프로젝트 생성
python .agents/skills/business-plan-writer/scripts/init_project.py

# 기존 프로젝트 목록
python .agents/skills/business-plan-writer/scripts/init_project.py --list

# 특정 프로젝트 로드
python .agents/skills/business-plan-writer/scripts/init_project.py --load {폴더명}
```

- 에이전트는 사용자 확인 없이 `--list`는 자동 실행 가능 (읽기 전용)
- `init_project.py` (생성 모드)는 반드시 사용자 확인 후 실행
